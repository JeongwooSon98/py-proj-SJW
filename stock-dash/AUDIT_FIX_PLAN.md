## 🔴 Phase A — 정의 버그 수정 (즉시, 백테스트 근거 불필요)

### A1. 볼린저 판정을 %B 정규화로 교체

**문제**: 밴드 하단 대비 **절대 2%** 완충을 쓰는데 밴드 폭은 종목 변동성에 비례한다. 저변동 종목(SPY)에서는 2%가 밴드폭의 81%를 먹어, "밴드 이탈(통계적 극단)"이 아니라 "20일선 아래"를 뜻하게 된다. 같은 `+2`점의 발화 빈도가 종목 간 5배 차이.

**수정 위치**: [src/StockDashboard.jsx](src/StockDashboard.jsx) `computeSignal` 4)번 블록 (현재 1006~1017행)

**Step**
1. `computeIndicators`에 `%B` 를 파생값으로 추가한다 (지표 파라미터는 여기 모은다는 규칙).
   ```js
   // 볼린저 %B — 밴드 내 상대 위치(0=하단, 0.5=중심, 1=상단). 밴드 폭으로 정규화돼
   //  종목 변동성과 무관하게 '극단'의 의미가 같아진다.
   out[i].pctB = (upper != null && lower != null && upper > lower)
     ? (r.close - lower) / (upper - lower) : null;
   ```
2. `computeSignal`에서 절대 % 조건을 `%B` 임계로 교체한다. 상수는 파일 상단 상수 블록으로 뺀다.
   ```js
   const BB_EXTREME = 0.05; // %B ≤ 0.05 = 하단 극단 / ≥ 0.95 = 상단 극단
   ```
   ```js
   if (last.pctB != null && last.pctB <= BB_EXTREME) { ...하단 +2 (레짐 시 +1)... }
   else if (last.pctB != null && last.pctB >= 1 - BB_EXTREME) { ...상단 −2 (레짐 시 −1)... }
   ```
3. **점수·만점·등급컷은 그대로**(+17 / −16, ±7/±3). 발화 조건만 정규화하는 것이므로 위계·정합성은 불변이다.

**검증**
- 발화율이 종목 간 균일해졌는지 확인 (기대: 3~13% 범위). 감사 때 쓴 스크립트와 동일한 방식으로 재측정.
- `npm run build` 통과.

**문서 동기화 (필수)**
- INDICATORS.md ①(볼린저 설명), 2장 점수표 4번 행의 판정 기준 `종가 ≤ 하단밴드 × 1.02 (2% 이내)` → `%B ≤ 0.05 (밴드 하단 5% 이내)` 로 수정.
- 왜 바꿨는지(변동성 정규화, 저변동 종목 남발 방지) 한 줄 근거를 문서에 남긴다.

### A2. 코드 ↔ 문서 ↔ 주석 불일치 정리

리팩터링 잔재로 **문서와 주석이 존재하지 않는 감점을 설명**하고 있다. 현재 동작(표시 전용)이 옳으므로 **문서/주석을 코드에 맞춘다.**

**Step**
1. **INDICATORS.md ⑬ (진입 위치)** — 같은 절 안에서 서로 모순된다. 아래를 **삭제**한다.
   - `- **감점만 한다(가산 없음)**: 점수를 0쪽으로만 수렴시켜…` 불릿 전체
   - 카드 표시 예시의 `, −1` 표기 → `🟡 지금 매수 진입 다소 확장 · 눌림 대기 유리 (MA20 +1.2×ATR)` 로 수정
   - "일절 반영하지 않습니다"는 코드와 맞으므로 **유지**.
2. **코드 주석 (현재 861~871행)** — `EXT_STRETCH_ATR`/`EXT_HOT_ATR` 위 주석의 "확신을 **감점**한다(가산 없음)", `→ −1`, `→ −2` 표기를 지우고 **"표시 전용 레벨 판정 임계"** 로 고쳐 쓴다.
3. **코드 주석 (현재 849~857행)** — `freshTiming` 위 주석의 "**점수를 절반 축소한다**" 삭제. 실제 코드는 `basePts`를 그대로 반환하며, 바로 아래 894~899행 주석이 이미 올바르게 설명하고 있으므로 중복·모순을 제거한다.
4. **RSI 문서 ↔ 코드 철학 불일치 명시** — INDICATORS.md ②의 "70 위에서 다시 70 아래로 내려오는 시점을 매도 신호로" 팁은 **코드에 구현돼 있지 않다**(코드는 `rsi > 70` 수준 판정). 둘 중 하나를 택한다.
   - (권장) 문서에 "⚠️ 현재 점수 체계는 수준(>70) 판정을 쓴다. 하향 크로스 방식으로의 변경은 Phase C4에서 백테스트 후 결정" 이라고 **현 상태를 정직하게 명시**.
   - 코드를 바꾸는 것은 가중치/조건 변경이므로 **Phase C로 미룬다.**

**검증**: INDICATORS.md의 모든 점수 언급이 `computeSignal` 실제 코드와 1:1 대응하는지 수기 대조. `npm run build`.

## 🟠 Phase B — 검증 인프라 (Phase C의 전제 조건)

**목표**: 백테스트가 "이 신호가 시장보다 나은가(알파)"를 재도록 고친다. 지금은 "강세장에 샀는가(베타)"를 재고 있다.
**⚠️ 이 Phase는 점수·임계값·만점을 일절 바꾸지 않는다.** 측정 도구만 고친다.

### B1. 기준선(baseline) / 초과수익 추가 — **최우선**

**문제**: `btReturns`가 절대 수익률을 집계한다. 표본이 최근 3년(**하락장 없음**) × 반도체·빅테크 15종목이라, "매수 신호 20일 승률 65%"가 신호의 능력인지 그냥 시장이 올랐기 때문인지 구분할 수 없다.

> ⚠️ **정정(2026-07-13)**: 이 계획서가 원래 "2022~2025 강세장"이라고 적었으나 **틀렸다.** `range=3y`는 오늘 기준 **2023-07~2026-07**이라 2022년 하락장이 표본에 **아예 없다.** 아래 "Phase B 결과" 참고.

**Step**
1. `backtestSeries`에서 종목별로 **전체 봉(워밍업 이후) 기준선**을 함께 집계한다. 즉 "아무 날에나 샀을 때"의 N봉 후 평균 수익률·승률.
   ```js
   // 기준선: 신호와 무관하게 워밍업 이후 모든 봉에서 진입했다고 가정한 성과.
   //  신호 성과가 이 값을 넘지 못하면 그 신호는 '시장 드리프트'일 뿐 알파가 아니다.
   events.push({ kind: "baseline", label: "기준선(전체 봉 진입)", dir: "buy", ret });
   ```
2. `aggregateBacktest`에 `baseline` 행을 별도로 반환하고, **매수 신호는 `buy` 기준선, 매도 신호는 부호 반전한 `sell` 기준선**과 비교한다.
3. UI 검증 패널에 각 신호 행마다 **`초과(vs 기준선)` 컬럼**을 추가한다. 평균수익률·승률 모두 차이값(%p)을 표시하고, **초과가 음수면 회색/경고 표시**로 "이 신호는 근거가 없다"를 눈에 보이게 한다.
4. 검증 패널 상단에 기준선 행을 고정 노출한다 (예: `기준선(전체 봉) · 5일 +0.4% / 승률 56% (n=9,800)`).

**검증**: 기준선 승률이 대략 52~60% 사이(강세장 표본이므로 50% 초과가 정상)로 나오는지 확인. 그보다 낮은 신호가 있으면 그 신호는 이후 Phase C 후보다.

### B2. 표본 신뢰도 정보 보강

**Step**
1. `aggregateBacktest`의 `stats`에 **중앙값(median)·표준편차·최악값(min)** 을 추가한다. "승률 70% / 평균 +0.5%"가 실은 가끔 −15% 터지는 신호일 수 있으므로.
2. **거래비용** 옵션을 추가한다. 왕복 기준 **미국 0.05%p, 한국 0.20%p**(거래세+슬리피지 근사)를 수익률에서 차감. 상수로 분리하고, 문서에 가정치를 명시한다.
3. 검증 패널 하단 "한계" 문구에 **겹치는 보유기간(overlapping windows) 경고**를 추가한다. 표시된 n은 독립 표본 수가 아니며, 워치리스트 15종목이 사실상 반도체·나스닥 단일 팩터라 **실질 독립 표본은 훨씬 적다**는 점을 명시.

### B3. ATR 청산 기준 백테스트 추가

**문제**: 카드는 **ATR 2배 손절 / 3배 목표(R:R 1.5)** 를 제시하는데, 백테스트는 고정 N봉 종가 청산이다. **화면에 표시하는 청산 규칙이 한 번도 검증된 적이 없다.**

**Step**
1. `btReturns` 옆에 `btAtrExit(rows, i, dir, atr)` 를 추가한다. 진입 봉 종가 기준, 이후 봉의 고가/저가로 **손절(2×ATR)·목표(3×ATR) 중 먼저 닿는 쪽**을 판정한다.
   - 같은 봉에서 둘 다 닿으면 **보수적으로 손절 우선**(look-ahead 방지).
   - 최대 보유 상한(예: 20봉) 내 미도달 시 종가 청산.
2. 결과를 **기대 R(= 평균 손익 ÷ 1R)** 과 **손절 도달률**로 집계해 별도 표로 노출한다.
3. 이 결과가 나쁘면 **ATR 배수(2×/3×)를 조정**할 근거가 된다 (표시 정보이므로 점수 체계와 무관, Phase B에서 바로 조정 가능).

### B4. 지연 진입 성과 (신선도 배지의 근거 만들기)

**문제**: 백테스트는 "신호가 처음 켜진 봉 종가 진입"만 잰다. 그러나 MACD는 5봉, 스토캐스틱은 3봉 동안 계속 채점되므로, **사용자는 신호 발생 3일째에 카드를 보고 산다.** 검증한 진입 시점과 실제 진입 시점이 다르다.

**Step**
1. `backtestSeries`에 **지연 진입(delay = 0, 1, 2, 3, 5봉)** 별 성과를 추가 집계한다.
2. UI에 "신호 발생 후 N봉 지연 진입 시 성과" 표를 추가한다.
3. 이 결과로 ⑫ 신선도 배지의 주황색 경고(1×ATR / 5봉)가 **실제로 의미가 있는지** 수치로 확인한다. 유의미하면 배지 문구에 근거를 넣고, 무의미하면 임계값을 조정한다.

### B5. 다이버전스 look-ahead 명시

**문제**: 스윙 피벗은 우측 3봉이 모여야 확정되므로, 백테스트가 마커를 읽는 시점에 약 3봉의 미래 정보가 섞인다(실전 동작은 정상, **백테스트만 낙관 편향**).

**Step**
1. 검증 패널의 `강세/약세 다이버전스` 행에 **⚠️ 배지**를 달고 "look-ahead 3봉 포함 — 이 수치는 근거로 쓸 수 없음" 툴팁을 붙인다.
2. (선택·권장) 백테스트 전용으로 **피벗 확정 지연을 반영한 인과적(causal) 다이버전스 탐지**를 구현한다. 봉 `i` 시점에서 `i-3` 이전 피벗만 사용하도록 제한하면 편향이 사라진다.

**Phase B 종료 조건**: 검증 패널이 **기준선 대비 초과수익**을 보여주고, 각 신호가 알파를 내는지 아닌지 눈으로 판별 가능한 상태.

1. 매도 평균회귀(RSI 과매수·볼린저 상단)의 **기본 점수를 −1로 낮추고**, **하락추세일 때만 −2** 로 올린다. (발동 빈도와 점수 의미가 일치)
2. 매수 평균회귀는 현행 유지(기본 +2, 위험 시 +1) — 위험 레짐이 13.6%로 드물게 발동하므로 현 구조가 의미를 갖는다.
3. 만점·등급컷·문서 ⑦ 서술("대칭 구조") 전면 재작성.

### C3. ⑩ 레짐 정합 보너스를 진짜 "천장만"으로 — ✅ **완료** (2026-07-15, U2)

**근거**: `score > 0`이면 +1을 더하므로 **score +2 종목이 +3이 되어 노출 임계를 넘었다.** 문서는 "천장만 1점 상향"이라 설명하지만 실제로는 **하단 컷을 1점 낮추는 marginal decider**였다. 근거는 **문서-코드 불일치(설계)이지 성과가 아니다.**

**한 일**: 게이트를 `score > 0` → **`score >= 3`**, `score < 0` → **`score <= -3`** 두 줄로 바꿈([computeSignal](src/StockDashboard.jsx) ⑩). 보너스 가산 **전** 점수가 이미 ±3을 넘은 종목에만 ±1을 더한다. 신호 정의·점수 크기(±1)·만점(+17/−16)·등급컷(±7/±3)·노출 임계(±3)·`btScoreBucket` 경계 **전부 불변**.

**검증** (같은 데이터로 HEAD 엔진 vs U2 엔진 비교):
- **기준선·지연 진입·개별 신호 행: 불변** (⑩ '정합' 2행 제외 — 그 신호의 발화 조건이 곧 이 변경의 대상이라 함께 감소: 추세장 1505→241, 횡보장 1043→463).
- **점수 구간: '있음(±3~6)' n만 감소**(매수 2638→2395, 매도 3532→3385 — +2→+3 유입이 사라짐), **'강함(±7)'은 불변**(82/134). = "천장은 유지, 하단 컷만 정합화"라는 설계 의도가 정확히 실현됨.
- 🔴 이 숫자(초과수익 변화)를 근거로 되돌리거나 확대하지 않는다. 이 변경은 성과와 무관하게 정당(문서-코드 일치).

### C4. 매도 쪽 비대칭 보정

**Step (B1 결과 확인 후)**
1. **매도 임계를 더 엄격하게**: 주식은 장기 우상향 드리프트가 있어 같은 |점수|의 매도 신호는 기대값이 구조적으로 나쁘다. 노출/등급 임계를 **−4 / −8** 로 올리는 안을 B1의 매도 기준선 대비 초과수익으로 검증 후 결정.
2. **거래량 신호 대칭화**: 현재 매수 전용(+1)이라 만점이 비대칭(+17/−16)이다. `거래량 급증 + 하락 = −1` 을 넣어 대칭화하거나, 매수 쪽도 제거한다. (B1에서 이 신호의 초과수익이 0 근처면 **제거가 정답**)
3. **RSI 하향 크로스 방식**(A2-4에서 미룬 것): `rsi > 70` 수준 판정 대신 **`prev.rsi > 70 && rsi <= 70`(70 아래로 복귀)** 방식을 B1로 비교 후 채택 여부 결정.

## 🟢 Phase D — 데이터 정합성·운영 (독립적, 언제든 가능)

### D1. 미완성 봉(repaint) 처리 — ✅ **완료** (2026-07-14, U1)

**문제**: `fetchSeries`가 일봉 마지막 봉 종가를 실시간 가격으로 덮어썼다. 장중에는 **미완성 봉**으로 신호를 채점하므로 신호가 켜졌다 꺼졌다 했다(repaint). `volume`은 갱신하지 않아 장중 **부분 거래량**이 쓰였고, 거래량 급증(+1)이 구조적으로 과소 발화했다.

> 🔴 **착수 후 정정 — 이 항목의 원래 문제 서술이 틀렸다.** 여기엔 "백테스트는 완성 봉만 쓰므로 검증한 것과 보고 있는 것이 다르다"고 적혀 있었는데, **백테스트도 완성 봉만 쓰지 않았다.** Yahoo는 장중에도 오늘 봉을 시계열에 넣고 그 **종가 자리에 현재가를 채워** 주므로, 앱 패널은 물론 **raw fetch를 쓰는 CLI(`npm run backtest`)까지 미완성 봉을 포함**하고 있었다. 실측 증거: 코드를 전혀 바꾸지 않고 장중에 CLI를 두 번 돌리자 초과수익 셀이 달라졌다(오늘의 진행 중 종가가 마지막 진입 몇 건의 청산가로 새어 들어감). 즉 **"검증한 것 ≠ 보는 것"이 아니라, 둘 다 오염돼 있었다.**

**한 일** — 채점 경로와 표시 경로를 분리했다 (→ [INDICATORS.md](INDICATORS.md) **1-b. 장중 표시 규칙**)
- `isLiveBar`: `meta.currentTradingPeriod.regular` + 마지막 봉 timestamp로 **미완성 봉을 판정**. KST 하드코딩 없음(미국·한국 거래시간이 다름). **판정 실패 시 보수적으로 '미완성'** 처리.
- `loadSeries` → **`fetchSeries`(채점·백테스트: 완성 봉만)** / **`fetchSeriesLive`(차트·현재가: 미완성 봉 포함, 단 신호 마커는 미표시)** 로 분리.
- **stale 보정은 살렸다** — `.KS`는 장 마감 후에도 그날 봉 `close`가 `null`로 와서 하루 뒤처진다. 이 보정은 **장중이 아닐 때만**(`!isInSession`) 적용되는 별도 분기다(합치면 미완성 봉이 되살아난다).
- **`.KS` KRX 장중 gap 차단** (2026-07-14 후속): `isInSession`(장중 판정)을 `isLiveBar`(미완성 봉 판정)에서 분리했다. `.KS`는 장중에 오늘 봉이 null로 빠져 `isLiveBar=false`가 되는데, stale 보정을 `!isLiveBar`로 걸면 그 틈으로 어제 봉이 실시간가로 덮여 **KRX 장중 repaint**가 난다. 조건을 `!isInSession`으로 바꿔 막고, 배지도 `isInSession` 기준으로 띄운다(그래야 `.KS`가 조용히 어제 점수로 채점되는 걸 사용자가 안다). **채점 엔진은 여전히 무변경**(HEAD 대비 백테스트 집계 동일). ⚠️ KRX 장중 실측은 못 해(장 마감) 합성 meta 5개 사분면으로 검증.
- **CLI도 같은 규칙으로** 통일(`isLiveBar`를 앱에서 그대로 꺼내 씀 — 복제본을 검증하지 않는다는 CLI 원칙 유지).
- 카드에 `🕐 장중 · 신호는 {날짜} 종가 기준` 배지.

**검증**
- **채점 로직 무변경 증명**: 동일 입력 데이터에 변경 전(`HEAD`)·변경 후 엔진을 각각 돌려 백테스트 집계 결과가 **완전히 동일**함을 확인(장중 데이터 변동이라는 변수를 제거한 비교).
- **CLI 결정성 회복**: 미완성 봉 제거 후에는 **장중에 연속 실행해도 출력이 바이트 단위로 동일**하다.
- 미완성 봉 제외의 표본 영향: 미국 종목 **정확히 1봉씩**(3771→3770봉), `.KS`는 불변. 기준선 진입 48,668 → 48,656건(**0.02%**). **2-b의 결론 수치는 그대로**(종합 매수 +3~6 초과 +0.1%p, +7↑ +0.4%p).

> ⚠️ **후속(미해결)**: 주봉(`1wk`)은 여전히 **진행 중인 이번 주 봉**으로 계산된다 — ⑥ 멀티 타임프레임(주봉 MA20/MA60)에 남아 있는 repaint 경로다. MA20/MA60은 느리게 움직여 실제 흔들림은 작지만 **0은 아니다.** 이걸 고치면 **점수가 바뀌므로**(설계 변경) 사용자 합의 없이 손대지 않았다.

### D2. 다이버전스 피벗 간격 제약 — ✅ **완료** (2026-07-15)

**문제**: `markDiv`가 직전 두 피벗을 비교하는데 **간격 제약이 없었다.** 5봉 떨어진 피벗과 80봉 떨어진 피벗을 똑같이 비교해 노이즈가 섞였다.

**한 일**: `markDiv`에 `gap = b − a`가 **`DIV_MIN_GAP`(5) ~ `DIV_MAX_GAP`(60)** 밖이면 건너뛰는 가드 추가. 근거는 **노이즈 제거(설계)**. 순전히 **필터(감소)**라 다이버전스 마커가 늘지 않는다.
- ⚠️ 이 제약은 `markDiv`(전체 시계열)에서만 걸린다 → **`computeSignal`의 스캔창(`DIV_LOOKBACK`=12)은 불변**이므로 `BT_SIG_WINDOW`(60)를 올릴 필요 없다(마커는 여전히 최근 12봉 안에서만 읽힌다).
- **검증**(HEAD vs 현재, 같은 데이터): 강세 다이버전스 428→385, 약세 839→787로 **감소만**. 다른 개별 신호·기준선 불변. 다이버전스는 타이밍 신호라 **지연 진입 n도 함께 감소**(9498→9403 등). 점수 구간은 다이버전스가 총점에 들어가므로 미세 이동.

### D3. 벤치마크 자기참조 제거 — ✅ **완료** (2026-07-15, 점수 무변경)

**문제**: `SPY`가 워치리스트이면서 동시에 미국 종목의 벤치마크다. SPY의 RS는 항상 0(신호 불가)이라 **SPY만 만점이 +16**이고, ⑦ 레짐 보정도 자기 자신 기준이라 순환한다.

**한 일**: 자기벤치(`benchmarkFor(sym) === sym`, 실질적으로 SPY)는 RS를 **명시적으로 제외**(rsShort/rsLong = null)하고 카드에 **"벤치마크 자신 — RS 미적용"** 노트를 띄운다.
- **점수 변화 없음**: RS는 원래 0 → 데드존(±3%p) → 신호 미발화였다. `null`이든 `0`이든 `computeSignal`의 RS 블록 결과가 동일(SPY 점수 불변 — 앱 스모크로 확인, 점수 2).
- ⚠️ **⑦ 레짐 보정은 의도적으로 건드리지 않았다**: 자기벤치의 레짐 = 시장 레짐이고, SPY의 역추세 신호를 시장 국면으로 누르는 건 SPY에도 타당하므로 순환이지만 무해하다. "레짐도 제외"는 **debatable한 점수 변경**이라 설계 명분이 약해 보류(원하면 별도 합의). SPY를 워치리스트에서 빼는 안은 **백테스트 표본이 바뀌므로** 채택 안 함.

### D4. 운영 리스크 — ✅ **완료** (2026-07-16)

- `corsproxy.io` 무료 프록시 + Yahoo 비공식 API가 **단일 장애점(SPOF)**이다. 죽으면 앱 전체가 멈춘다.
  - **폴백 프록시는 채택 못 함**: 브라우저 실측(2026-07-16) 결과 allorigins·thingproxy·codetabs가 모두 Yahoo에 대해 동작하지 않았고, 성공한 것은 corsproxy.io 변형뿐이라 **독립적 폴백이 되지 못했다**(같은 제공자 = 여전히 SPOF). 죽은 폴백을 넣는 건 해로워 넣지 않았다.
  - **대신 채택**(audit이 명시한 "또는 명확한 사용자 안내"): `fetchChart`에 **일시적 실패 1회 재시도**(600ms 지연) + RecommendPanel·백테스트·차트 **에러 문구를 실행 가능하게**("프록시/네트워크 문제일 수 있습니다 — 새로고침"). SPOF 위험 자체는 코드 주석·이 문서에 명시.
- Yahoo `quote` 배열은 **분할은 반영, 배당은 미반영** → INDICATORS.md 2-b "한계"에 명시(15년 표본에서 누적 편향이 더 크나 **초과수익엔 거의 무영향**임을 강조).
