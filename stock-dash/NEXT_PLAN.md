# 🛠 실사용 전환 계획서 — U1~U4

작성: 2026-07-14 · 선행: [REFRAME_PLAN.md](REFRAME_PLAN.md) R1~R4 완료(라벨·문구 격하 + 정직성 고지)

## 이 계획의 전제 (읽지 않으면 방향이 틀어진다)

15년 검증 결과 **종합 점수의 초과수익은 +0.1%p** — 아무 날에나 산 것과 구분되지 않는다([INDICATORS.md](INDICATORS.md) 2-b). R1~R4에서 "추천"이라는 **주장**은 이미 걷어냈다. 남은 질문은 하나다:

> **그러면 이 대시보드를 실제 매매에 어떻게 써먹는가?**

답: **"뭘 살지"를 앱에 묻지 않는다. "어디를 볼지"를 묻는다.** 앱의 역할을 *선별기(selector)* 에서 *관측·기록 도구(scanner + journal)* 로 바꾼다. 이 역할이라면 알파가 없어도 가치가 성립한다.

### 이번 계획이 **하지 않는** 것

| 안 하는 것 | 이유 |
|---|---|
| 지표 추가 | 15년·양 레짐·거래비용 차감으로 안 나온 우위는 지표를 더 얹어서 나오지 않는다 |
| 가중치 튜닝 · 임계값 최적화 | 근거가 없다. 나온 것처럼 보이면 그건 **과최적화** |
| MACD 부호 반전 | 단일 신호·다중검정 미보정·생존편향 미보정 |
| "초과수익 0이니 그 신호 삭제" | **'우위 없음'은 '해롭다'가 아니다.** 제거는 성과가 아니라 **설계 근거**로만 |

## 실행 순서와 그 이유

```
U1 (완성 봉 채점)  →  U2 (⑩ 보너스 정합화)  →  U3 (변화 감지)  →  U4 (매매일지)
   정합성 버그          문서-코드 불일치         관측 도구화        측정 도구화
```

**U1이 반드시 먼저**다. 지금 화면의 점수는 장중에 **미완성 봉**으로 채점된 값이라 흔들린다(repaint). 그 위에 U3(변화 감지)·U4(매매일지)를 얹으면 **흔들리는 숫자를 기록하게 된다** — 어제 대비 델타가 신호 변화인지 장중 노이즈인지 구분할 수 없고, 일지에 남는 "진입 시점 점수"도 종가에 바뀌어 버린다.

## U1. 완성 봉 기준 채점 (구 D1 · **최우선**)

### 문제

[fetchSeries](src/StockDashboard.jsx#L168-L176)가 일봉 **마지막 봉의 종가를 실시간 가격으로 덮어쓴다.**

```js
if (interval === "1d" && rows.length) {
  const price = result.meta?.regularMarketPrice;
  const last = rows[rows.length - 1];
  if (price != null && price !== last.close) {
    last.close = price;                       // ← 장중이면 미완성 봉이 된다
    last.high = Math.max(last.high, price);
    last.low = Math.min(last.low, price);
  }                                            // ← volume은 갱신 안 함
}
```

결과:

1. **신호가 켜졌다 꺼졌다 한다(repaint).** 오후에 본 "MACD 골든크로스"가 종가에 사라질 수 있다. **"근거 확인"의 전제가 깨진다.**
2. **`volume`을 갱신하지 않아** 장중에는 **부분 거래량**으로 채점된다 → 거래량 급증(+1)이 구조적으로 **과소 발화**한다.
3. **백테스트는 완성 봉만 쓴다.** 검증한 것과 화면에서 보는 것이 **다른 물건**이다.
4. **앱 내 백테스트 패널도 오염된다** — [loadBacktest](src/StockDashboard.jsx#L395)가 같은 `fetchSeries`를 쓰므로 **마지막 봉이 실시간 가격으로 덮여** 있다. (CLI `scripts/backtest-cli.mjs`는 Yahoo를 **직접** 호출하므로 영향 없음 → 그래서 둘의 결과가 미세하게 다를 수 있다.)

### ⚠️ 함정 — 이 코드에는 목적이 **두 개** 섞여 있다

주석이 말하듯 원래 목적은 **(a) stale 시계열 보정**(일부 `.KS` 종목은 일봉이 며칠 뒤처져 묵은 종가로 지표가 계산됨)이고, **(b) 장중 미완성 봉 최신화**는 그 부작용이다. **덮어쓰기를 그냥 지우면 (a)가 되살아나 묵은 종가로 채점하게 된다.** 둘을 분리해야 한다.

### 설계 (권장안 — 보수적)

**채점 경로와 표시 경로를 분리한다.**

| 용도 | 데이터 |
|---|---|
| **지표 계산·점수 채점** (`computeIndicators` → `computeSignal`) | **완성 봉만.** 장중이면 마지막(미완성) 봉을 **잘라낸다** |
| **차트·현재가 표시** | 지금처럼 실시간 가격 반영 (미완성 봉은 시각적으로 구분) |
| **백테스트** (앱 패널 · CLI 둘 다) | **완성 봉만.** 마지막 봉 미완성이면 제외 |

**미완성 봉 판정** — `meta.regularMarketPrice !== last.close` **만으로는 stale과 구분할 수 없다**(둘 다 불일치를 만든다). Yahoo `meta`의 아래 필드로 판정할 것:

- `meta.currentTradingPeriod.regular.{start,end}` (epoch) 와 `meta.regularMarketTime` 비교 → **장중이면 미완성**
- 또는 마지막 봉의 timestamp가 **오늘 거래일**이고 현재 시각이 정규장 종료 전이면 미완성
- ⚠️ **한국 종목(`.KS`)과 미국 종목의 거래시간이 다르다.** 하드코딩한 KST 시각으로 판정하지 말고 **meta가 주는 값**을 쓸 것 (타임존 버그가 조용히 들어간다).
- 판정에 실패하면(필드 없음 등) **보수적으로 "미완성"으로 간주**해 잘라낸다. 흔들리는 점수보다 하루 늦은 점수가 낫다.

**stale 보정은 살린다** — 마지막 봉이 **오늘이 아니고** 시계열이 뒤처진 경우에는 기존 덮어쓰기 유지(또는 최소한 현재가를 차트·현재가 표시에 반영). "장중 미완성"과 "시계열 지연"을 코드에서 **다른 분기**로 다룰 것.

### UI

- 카드에 **`🕐 장중 — 신호는 전일 종가 기준`** 배지 (장중일 때만). 사용자가 "왜 점수가 안 움직이지?"를 묻지 않게 한다.
- 차트의 미완성 봉은 그대로 그리되(현재가 확인용), 신호 마커는 찍히지 않는다.

### 검증

- `npm run build` 통과.
- **`npm run backtest` 출력은 완전히 동일해야 한다** — CLI는 원래 raw fetch라 이 변경의 영향을 받지 않는다. **달라지면 채점 로직을 잘못 건드린 것이다.**
- **장 마감 후**에 앱을 열면 점수가 변경 전과 **동일**해야 한다(완성 봉 = 실시간 종가).
- **장중**에는 새로고침을 반복해도 점수·배지가 **흔들리지 않아야** 한다(이게 이 작업의 목적이다).
- 앱 백테스트 패널과 CLI 결과가 **일치**하는지 확인(마지막 봉 오염 제거로 오히려 가까워져야 한다).

### 문서

INDICATORS.md에 **"장중 표시 규칙"** 절 신설: 신호는 **직전 완성 봉** 기준이고, 장중 현재가는 표시 전용임을 명시. AUDIT_FIX_PLAN.md D1 체크.

## U2. ⑩ 레짐 정합 보너스 정합화 (구 C3 / REFRAME R5-2)

### 문제 — **성과가 아니라 문서-코드 불일치가 근거다**

[StockDashboard.jsx:1169](src/StockDashboard.jsx#L1169) (매도 대칭은 1186행)

```js
if (last.adx != null) {
  const trendRegime = last.adx >= 25;
  const rangeRegime = last.adx < 20;
  if (score > 0) {                       // ← 게이트가 '부호'뿐이다
    if (trendRegime && hasTrendBuy)  addBuy("추세장 추세신호 정합", 1, ...);   // +1
    else if (rangeRegime && hasMeanRevBuy) addBuy("횡보장 평균회귀 정합", 1, ...);
  } else if (score < 0) { /* 매도 대칭 −1 */ }
}
```

`addBuy`/`addSell`이 `score`에 직접 가산하므로, **점수 +2인 종목이 +3이 되어 노출 임계를 넘는다.** 고칠 곳은 **게이트 조건 두 줄**(`score > 0` → `score >= 3`, `score < 0` → `score <= -3`)이다. 신호 정의·점수 크기(±1)는 건드리지 않는다.

**문서([INDICATORS.md](INDICATORS.md) ⑩)는 "천장만 1점 상향"이라고 설명하지만, 코드는 하단 컷을 1점 낮추고 있다.** 즉 이 보너스가 **목록에 뜨는 종목 구성 자체를 바꾼다** — 다른 신호가 +2밖에 못 모은 종목이 ADX 정합 하나로 노출된다. 이건 실사용에서 사소하지 않다.

> ⚠️ **이 변경의 근거는 "백테스트에서 이 신호의 초과수익이 −0.3%p라서"가 아니다.** 그건 부당한 근거다(‘우위 없음’ ≠ ‘해롭다’). 근거는 **문서가 말하는 설계와 코드가 다르다**는 것 하나뿐이다. 이 구분을 흐리지 말 것.

### 설계 (권장안 — footprint 최소)

**보너스를 "이미 노출 임계(±3)를 넘은 종목"에만 적용한다.**

- 보너스 **적용 전** 점수 `|score| ≥ 3` 일 때만 ±1을 더한다.
- **만점 +17 / −16 불변**, 등급컷(±7/±3)·노출 임계(±3) **숫자 불변**, `btScoreBucket` 경계 **불변**.
- 문서의 "천장만 올린다"는 서술이 **비로소 참이 된다.**

**대안(비권장)**: 보너스 완전 제거 → 만점 **+16 / −15** 로 바뀌고 등급컷·`btScoreBucket`·문서 만점 표기를 **전부 재계산**해야 한다. 고칠 것이 훨씬 많은데 얻는 것은 같다.

### ⚠️ 이건 **점수 변경**이다 — 검증 규칙이 U1과 다르다

`npm run backtest` **출력이 달라진다. 달라지는 게 정상이다.** 그러니:

1. **착수 전에 반드시 기준값을 저장**: `npm run backtest > /tmp/bt_before.txt`
2. 변경 후 diff를 뜨고 **무엇이 왜 바뀌었는지 설명할 수 있어야 한다.**
   - **① 개별 신호 행: 바뀌면 안 된다** (신호 정의를 안 건드렸으므로). 바뀌었다면 실수한 것이다.
   - **② 종합 점수 구간 행: 바뀌는 게 맞다** (점수 분포가 달라지므로 n·수치 이동).
3. 🔴 **바뀐 초과수익 숫자를 근거로 이 변경을 되돌리거나 확대하지 말 것.** 좋아지면 "역시 옳았다", 나빠지면 "되돌리자" — **둘 다 p-hacking이다.** 이 변경은 성과와 무관하게 정당하다(문서-코드 일치). 결과는 **기록만** 한다.

- 점수 모달 문구(만점·등급컷은 그대로지만 ⑩ 설명이 있다면 수정)

## U3. 변화 감지 (diff) — "오늘 바뀐 것"

### 왜

지금 앱은 **"오늘 점수가 몇 점"** 만 보여준다. 그런데 실사용에서 기회를 놓치는 지점은 **"어제 없던 신호가 오늘 켜졌다"** 이다. 15개 종목을 매일 눈으로 훑는 대신 **바뀐 것만** 본다. **이건 알파를 주장하지 않고도 성립하는 가치다** — "변화"는 예측이 아니라 사실이다.

### 설계

- **스냅샷**: `localStorage`에 마지막 조회 결과를 저장 — `{ barDate, symbol → { score, signalLabels[] } }`.
  - ⚠️ 키는 **완성 봉의 날짜(`barDate`)** 다. **U1이 선행돼야 하는 이유** — 장중 미완성 봉을 스냅샷으로 저장하면 델타가 하루 종일 요동친다.
  - 같은 `barDate`면 덮어쓰지 않는다(하루에 여러 번 열어도 기준이 고정).
- **표시**:
  - 카드 배지에 **`🆕`** (직전 스냅샷에 없던 신호) 표시
  - 점수 칩에 델타 (`+5점 ▲3`)
  - 패널 상단에 **"어제 대비 바뀐 것"** 요약: 새로 노출된 종목 / 목록에서 빠진 종목
- **문구 규칙**: "새 기회" 같은 말을 쓰지 말 것. **"어제 없던 신호"** 처럼 **사실만** 서술한다(R1~R4에서 세운 원칙).

### 주의

- 첫 실행·스냅샷 없음 → **델타를 표시하지 않는다**(0으로 표시하지 말 것).
- 주말·휴장으로 `barDate`가 안 바뀌면 델타는 **전 거래일 대비**다. 라벨에 기준 날짜를 함께 표기.

## U4. 매매일지 — **이게 제일 중요하다**

### 왜

15년 백테스트가 말해주는 건 **"이 10개 신호를 기계적으로 합산하면 우위가 없다"** 이지, **"당신의 판단에 우위가 없다"** 가 아니다. 이 화면을 보고 실적·뉴스·업황을 얹어 내리는 **재량 판단은 한 번도 측정된 적이 없다.** 그걸 재는 유일한 방법은 **기록**이다.

**기록이 없으면 6개월 뒤에도 "이 대시보드가 도움이 됐는지" 알 수 없다.** 이 프로젝트에서 아직 만들지 않은 유일한 진짜 도구다.

### 설계 (최소 스펙)

- **진입 기록**: 카드에서 `기록` 버튼 → `{ symbol, date, side(buy/sell), price, score, signals[](그때 켜져 있던 라벨), entryLevel, memo }`
  - **점수와 신호 라벨을 그때 그대로 박제**하는 것이 핵심이다(나중에 재계산하면 안 된다).
- **청산 기록**: `{ exitDate, exitPrice, reason(목표/손절/시간/재량) }` → 수익률 · **R 배수**(진입 시 ATR 손절폭 기준) · 보유일 자동 계산
- **집계 뷰**: 내 매매의 평균/중앙 수익률 · 승률 · 기대 R **vs 같은 기간 기준선**(아무 날에나 산 것)
  - **백테스트와 같은 언어(초과수익)로 표시할 것.** 절대 수익률만 보면 또 베타를 알파로 착각한다.
- **저장·백업**: `localStorage` + **JSON export/import 필수**. 브라우저 하나 날리면 데이터가 사라진다.

### 해석 규칙 (UI에 박아 둘 것)

- **표본수(n)와 기간을 항상 함께 표시.** n < 30이면 **"표본 부족 — 결론 내지 말 것"** 문구를 띄운다.
- 겹치는 보유기간·소수 종목 상관 때문에 **실질 독립 표본은 n보다 적다**(2-b의 한계와 동일).
- **사후에 합격 기준을 만들어 붙이지 말 것.** "이번 분기만 빼면 좋았다"류의 해석 금지.

### 🔴 U1 중 발견 — 계획서의 전제 하나가 틀렸다

이 계획서는 "**CLI는 raw fetch라 미완성 봉의 영향을 안 받는다 → 출력이 동일해야 한다**"를 U1의 검증 기준으로 삼았다. **틀렸다.** Yahoo는 장중에도 오늘 봉을 시계열에 넣고 그 **종가 자리에 현재가를 채워** 주므로, **CLI도 미완성 봉을 포함**하고 있었다.

- **증거**: 코드를 전혀 바꾸지 않고 장중에 `npm run backtest`를 두 번 돌리자 초과수익 셀이 달라졌다. 오늘의 진행 중 종가가 **마지막 진입 몇 건의 청산가**로 새어 들어갔기 때문이다.
- 즉 D1의 문제 서술("백테스트는 완성 봉만 쓴다")도 사실이 아니었다 — **앱과 CLI 둘 다 오염돼 있었다.**
- **조치**: 계획서 설계표("백테스트 앱·CLI 둘 다 = 완성 봉만")대로 CLI도 통일. 근거는 **성과가 아니라 정합성**이다.

## 🔴 이 계획 전체에 걸친 절대 규칙

3. **"추천·적극 매수·오늘 매수 적합" 같은 처방 문구를 다시 넣지 않는다** (R1~R4에서 의도적으로 제거).
4. 코드를 바꾸면 **[INDICATORS.md](INDICATORS.md)를 반드시 동기화**한다.
5. 주석·문서·응답은 **한국어**.
